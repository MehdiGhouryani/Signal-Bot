# فاز ۵ — سیستم مالی (بند ۱۱، ۱۲، ۱۳ نیازمندی‌ها)

بزرگ‌ترین فاز تا الان. سه بخش مجزا: سخت‌سازی Prize Pool، سیستم Reward مستقل، و
Donate مستقیم به یک Alpha Caller. هر بخش جدا پیاده‌سازی، تست، و شبیه‌سازی شد.

## 🐛 یه باگ واقعی که در حین کار پیدا شد: چک وضعیت پرداخت هیچ‌وقت کار نمی‌کرده

قبل از ساخت وب‌هوک، مسیر داده‌ی پرداخت رو دنبال کردم و متوجه شدم:
`order_id` تولید می‌شد و به NOWPayments فرستاده می‌شد، ولی **هیچ‌جا ذخیره نمی‌شد**.
`payment_id` هم هیچ‌وقت ست نمی‌شد. یعنی دکمه‌ی «🔄 چک وضعیت پرداخت» از همون فایل
اولیه، همیشه با `payment_id=None` صدا زده می‌شده و شکست می‌خورده (بی‌صدا، چون
`check_payment` خطا رو می‌گرفت و «unknown» برمی‌گردوند). این باگ توی هر ۵ فاز قبلی
هم بدون تغییر مونده بود چون کسی مستقیم رو مسیر دادهٔ پرداخت دست نذاشته بود.

**رفع شد:** حالا `order_id` توی دیتابیس ذخیره می‌شه (ستون جدید)، و `payment_id`
هروقت از NOWPayments دریافت بشه (چه از وب‌هوک، چه در آینده) ذخیره می‌شه. اگه
`payment_id` هنوز معلوم نباشه، دکمه‌ی چک دستی صادقانه می‌گه «هنوز اطلاعی نداریم»
به‌جای اینکه وانمود کنه چک کرده.

## ✅ ۱. وب‌هوک IPN (تأیید خودکار پرداخت)

یه سرور کوچیک `aiohttp` که کنار polling ربات، روی یه پورت جدا بالا میاد (فقط اگه
`NOWPAYMENTS_IPN_SECRET` رو پر کنی — وگرنه غیرفعال می‌مونه، بدون کرش).

**امضا:** طبق مستندات رسمی NOWPayments (HMAC-SHA512 روی JSON با کلید مرتب‌شده)
پیاده‌سازی و با ۵ سناریو (امضای معتبر، دستکاری‌شده، سکرت غلط، خالی، ترتیب کلید
متفاوت) تست شد.

**Routing:** هر invoice یه `order_id` خودتوصیف‌گر داره (`pool_...` برای استخر
عمومی، `support_...` برای حمایت مستقیم) — وب‌هوک از روش می‌فهمه کدوم جدول رو
آپدیت کنه.

**تست شد (با HTTP واقعی، نه فقط تابع):** IPN معتبر برای هر دو نوع، امضای نامعتبر
(۴۰۱)، JSON خراب (۴۰۰، نه کرش)، order_id ناموجود (۲۰۰ + لاگ، نه خطا)،
idempotency (IPN تکراری برای پرداخت already-paid دوباره اطلاع‌رسانی نمی‌کنه).

**⚠️ محدودیت صادقانه:** نتونستم این رو در برابر یک IPN واقعی از سرور NOWPayments
تست کنم (سندباکس من نه به NOWPayments نه به تلگرام دسترسی داره). فقط منطق داخلی
تست شده. **قبل از اعتماد کامل، با یک پرداخت واقعی کوچیک تستش کن.** دکمه‌ی چک
دستی به‌عنوان fallback هنوز کار می‌کنه و مستقل از وب‌هوکه.

**⚠️ نیاز به زیرساخت:** این وب‌هوک برای کارکرد واقعی نیاز به یه **آدرس عمومی
HTTPS** داره (دامنه + ریورس‌پروکسی، یا سرویسی مثل ngrok برای تست). این یه تصمیم
هاستینگ/دامنه‌ست که از دست من خارجه — راهنمای `.env.example` رو ببین.

## ✅ ۲. سیستم Reward مستقل (بند ۱۲)

جدول جدید `rewards`، کاملاً جدا از `prize_pool`/`prize_seasons` — یعنی هیچ اثری
روی منطق تقسیم جایزه‌ی پایان دوره نداره. فعلاً فقط اعطای دستی توسط ادمین (از
همون «مدیریت کاربران» → «🎁 اعطای پاداش» → مبلغ + دلیل)، ولی ساختارش آماده‌ست
که بعداً یه موتور خودکار (بند ۱۲ گفته «در آینده») بدون تغییر schema همین جدول
رو پر کنه. کاربر از «🎁 پاداش‌های من» تو منوی اصلی، تاریخچه‌شو می‌بینه.

## ✅ ۳. Donate مستقیم به Alpha Caller (بند ۱۳)

جدول جدید و کاملاً مجزا `caller_donations` (نه ستون اضافه به `prize_pool`، چون
مفهوماً دو جریان پول متفاوتن). دکمه‌ی «💝 حمایت از {نام}» رو کنار هر سیگنال تو
فید عمومی «سیگنال‌های فعال» می‌بینی (به‌جز رو سیگنال خودت). مبلغ رو انتخاب می‌کنی
→ فاکتور NOWPayments → بعد از تأیید (وب‌هوک یا چک دستی)، هم به تو هم به همون
تحلیلگر پیام تأیید می‌ره.

## 📁 خلاصه فایل‌های جدید

`db/caller_donations_repo.py`, `db/rewards_repo.py`, `services/ipn_signature.py`,
`web/ipn_server.py`, `handlers/support.py` — به‌علاوه ستون `order_id` روی
`prize_pool` و بازنویسی جزئی `handlers/payments.py` برای رفع باگ order_id.

## ✅ شبیه‌سازی مسیرهای کامل که انجام شد

- **کاربر عادی:** واریز به استخر عمومی (بدون تغییر) ✅ | حمایت مستقیم از یه
  تحلیلگر از فید عمومی ✅ | دیدن پاداش‌های خودش ✅
- **ادمین:** اعطای پاداش دستی به یه کاربر ✅ | دیدن هم واریزهای استخر هم
  حمایت‌های مستقیم در یک صفحه ✅ | فرمت غلط ورودی پاداش رد می‌شه بدون کرش ✅
- **سعی در حمایت از خود:** رد می‌شه با پیام مناسب ✅
- **وب‌هوک:** هر دو نوع تراکنش (pool/support) درست routing و idempotent ✅

## ✅ تست‌های انجام‌شده (خلاصه کلی)

Syntax + import کامل، ساخت Application با ۱۳ هندلر (شامل post_init/post_shutdown
برای وب‌هوک)، ۶۰ نوع callback_data بدون تداخل، رفع باگ order_id تأیید شد،
۵ تست امضای IPN، ۶ تست HTTP واقعی وب‌هوک (با aiohttp TestClient)، فلوی کامل
Reward و Caller Donation با mock، و رگرسیون کامل فاز ۱ تا ۴ + گروه.

**آنچه به‌خاطر محدودیت شبکه‌ی سندباکس تست نشد:** فراخوانی واقعی API نوپیمنتس
(`create_invoice`/`check_payment`) و دریافت IPN واقعی — این‌ها با mock تست شدن،
نه با اتصال واقعی. حتماً قبل از استفاده‌ی جدی، با پرداخت واقعی کوچیک تست کن.
