# فاز ۳ — بازطراحی ثبت سیگنال + فید عمومی (بند ۴ و ۶)

## ✅ چیزهایی که اضافه/تغییر کرد

### ۱. ویزارد ۷مرحله‌ای قبلی حذف شد — جاش دو مسیر خیلی کوتاه‌تر:

**📸 Full Signal** — دقیقاً طبق سند: «کاربر بتواند تنها تصویر سیگنال را ارسال کند».
فقط عکس رو می‌فرسته (با کپشن اختیاری برای اسم کوین/جهت) → تمام. اگه کپشن بده،
ربات با یک heuristic ساده سعی می‌کنه اسم کوین (کلمه اول لاتین) و جهت (کلمه
long/short یا خرید/فروش) رو ازش تشخیص بده؛ اگه نتونه، سیگنال بازم بدون مشکل
ثبت می‌شه — چیزی اجباری نیست.

**⚡ Fast Call** — دقیقاً طبق سند: «معرفی سریع قرارداد یا توکن بدون Entry و TP».
فقط اسم کوین/توکن (یا آدرس قرارداد) رو می‌نویسه، جهت رو هم می‌تونه بزنه یا رد کنه
(⏭ رد کردن جهت) → تمام. ۲ پیام کل ماجراست.

هر دو مسیر از همون pipeline قبلی برای بررسی (Admin/VIP Helper تأیید یا رد می‌کنن)
و ثبت نتیجه (دستی توسط ادمین) عبور می‌کنن — چیزی در اون بخش عوض نشده.

### ۲. فید عمومی «سیگنال‌های فعال» (بند ۶ — قبلاً کلاً وجود نداشت)
دکمه جدید تو منوی اصلی، برای همه کاربران: سیگنال‌های تأییدشده و باز رو با
👤 نام ثبت‌کننده + 🎖 رول‌اش نشون می‌ده (Alpha Score قراره فاز ۴ اضافه بشه چون
هنوز پیاده نشده). برای Full Signal، عکس واقعی رو هم می‌فرسته.

### ۳. try/except با لاگ‌نویسی فشرده (طبق درخواستت)
یه ماژول جدید `services/notify.py` ساختم: `safe_send_message` /
`safe_send_photo` / `safe_send_document`. همه‌ی جاهایی که قبلاً
`try: ... except Exception: pass` بی‌صدا بودن (تو admin.py، signals.py،
payments.py، jobs/scheduled.py) با این توابع جایگزین شدن — حالا هر شکست ارسال
پیام (کاربر بلاک کرده، چت پیدا نشد و...) با `logging.warning` یک‌خطی و بامعنی
ثبت می‌شه، به‌جای قورت‌دادن کامل خطا. `services/payments.py` هم دقیق‌تر شد:
به‌جای `except Exception`، مشخصاً `requests.RequestException` و `ValueError`
(برای JSON نامعتبر) رو می‌گیره.

### ۴. یه باگ واقعی که تست پیدا کرد و رفعش کردم
تابع heuristic تشخیص کوین از کپشن اولش از `str.isalpha()` استفاده می‌کرد که
برای حروف فارسی هم `True` برمی‌گردونه (نه فقط لاتین) — یعنی اگه کسی کپشن فارسی
می‌نوشت («یه تحلیل خوب...»)، ربات اشتباهی کلمه اول فارسی رو به‌عنوان «اسم کوین»
ثبت می‌کرد. با اضافه‌کردن چک `isascii()` رفعش کردم. این دقیقاً همون چیزیه که
تست عملی قبل از تحویل باید گیرش بندازه — و گرفت ✅.

## 📁 تغییرات فایل/دیتابیس

- `signals` جدول: دو ستون جدید `signal_type` ('full'/'fast') و `photo_file_id`.
  ستون‌های `entry`/`stop_loss`/`take_profit` دیگه اجباری نیستن (از قبل هم NOT
  NULL نبودن، فقط الان واقعاً NULL می‌مونن برای سیگنال‌های Fast/Full-بدون-عدد).
- `db/signals_repo.py`: `insert_signal()` بازنویسی شد (پارامترهای اختیاری)،
  تابع جدید `get_public_feed()`.
- `handlers/signals.py`: کامل بازنویسی شد. `photo_handler` جدید (MessageHandler
  مجزا برای عکس، در `main.py` ثبت می‌شه).
- کیبوردهای منسوخ حذف شدن: `timeframe_kb`, `market_kb`, `leverage_kb`,
  `skip_desc_kb` (دیگه استفاده‌ای ندارن).

## ✅ تست‌هایی که قبل از تحویل انجام شد

1. Syntax + import کامل
2. ساخت واقعی Application با پترن‌های جدید + MessageHandler عکس (۱۰ هندلر)
3. **۵۴ نوع callback_data** — بدون تداخل یا جاافتادگی
4. Heuristic پارس کپشن: خالی/انگلیسی‌با‌جهت/فارسی‌با‌کلمه‌انگلیسی/فارسی‌خالص —
   **یک باگ واقعی همینجا پیدا و رفع شد** (بالا توضیح داده شد)
5. ثبت واقعی Full Signal (با عکس) و Fast Call (بدون جهت) → تأیید → فید عمومی
   (رول نمایش داده می‌شه) → ثبت نتیجه → امتیازدهی صحیح برای هر دو نوع
6. `my_signals_text` برای هر دو نوع سیگنال بدون خطا
7. رگرسیون کامل فاز ۱ و ۲ (پروفایل، لیدربورد، استخر جایزه، تابلوی افتخار،
   VIP Helper، محدودیت رول) — همه سالم
8. بررسی دستی: هیچ `except Exception: pass` بی‌لاگ توی کل پروژه باقی نمونده
   (فقط یک استثنا: آپگرید ستون دیتابیس که «ستون از قبل هست» رو عمداً و با
   کامنت توضیح‌دار نادیده می‌گیره، ولی خطاهای دیگه رو لاگ می‌کنه)

مثل همیشه: این تست‌ها منطق/دیتابیس/routing رو تضمین می‌کنن، ولی جایگزین تست
واقعی ارسال عکس از اپ تلگرام نیستن (چون سندباکس من به سرورهای تلگرام دسترسی
نداره) — لطفاً حتماً یه بار خودت از طریق اپ عکس بفرست و مطمئن شو.
